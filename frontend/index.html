Commit updates

Every client whose running checkout is behind origin/master, in one place — select some or all and update them in one run instead of opening each instance. Parallel across VPSes, one at a time per VPS (several simultaneous docker builds on one small box is how an update becomes an outage). Each update is the exact same guarded pipeline as the per-client Deploy button: git pull --ff-only → build → container-name collision precheck → restart, each stage gating the next.

Fleet tests

        
Run the full check suite against any set of clients in one go — the 9-check smoke
        suite (loopback + public health, .env sanity — structure, no default admin
        password, backup passphrase present; values never leave the box — TLS certificate, /config sanity,
        admin API, embed CSP, a real chat round-trip that proves the LLM key end to end,
        backup timer) plus live site_config.yaml validation. All checks are read-only; the
        chat round-trip burns a few LLM tokens per client — that's the cost of actually proving it's
        alive. Parallel across VPSes, one client at a time per VPS; the chat check alone can take ~30s
        per client.

Backups

        
Installs (or repairs) the nightly encrypted-backup timer on each
        selected client's VPS — correctly-named units pointing at that client's real checkout — then
        proves it by running a real backup immediately and checking a fresh encrypted
        archive appears on disk (a timer that fires isn't a backup; only the archive is). Uses the same
        selection as the table above. Idempotent — safe to re-run any time; needs a
        BACKUP_PASSPHRASE in the instance's .env (the wizard always generates
        one; otherwise set it via Credentials first).

New client

      
Clones the repo, picks a free port, generates a bootable placeholder .env + starter site_config.yaml, builds and boots it, wires it into the shared Caddy, and registers it here — all over SSH. You only need to know the deploy name and the hostname; everything else (real API keys, business content) you fill in afterward via Credentials and the admin panel.
